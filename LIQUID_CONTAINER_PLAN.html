LiquidContainer Runtime Architecture

Status: PLANNING COMPLETE

Goal: Replace mock orchestration with a secure, Docker-based real-time agent runtime supporting multiple Agent SDKs.

1. Core Concept: The "LiquidContainer"

The LiquidContainer is a standardized Docker environment designed to be the universal runtime for any AI agent in the LiquidCrypto ecosystem.

Design Principles

  • Agnostic: The infrastructure layer runs code, not specific models. NO vendor lock-in in the container name.
  • Feature Parity: Mirrors the host's development environment (Bun, Node, Git) for seamless script execution.
  • Real-Time: Utilization of a "Warm Pool" for near-instantaneous (< 500ms) availability.

2. Comparison of Agent SDKs (The Ecosystem)

We analyzed SDKs from major providers (Claude, OpenAI, Google, MiniMax) and identified two distinct architectural patterns:

A. "Local Brain"

Claude Agent SDK

Logic runs inside the container.

B. "Remote Brain"

OpenAI, Google, MiniMax

Logic runs in Cloud; Container is "Hands".

Feature Claude Agent SDK OpenAI / Google / MiniMax
Logic Location Local (Container) Cloud
Tool Execution Local (Built-in) Local (Callback)
Role of Container The Agent Itself The Bridge / Hands

3. Integration Strategy: "The Unified Bridge"

To support both patterns, LiquidContainer will support dynamic Runtime Profiles selected by the Orchestrator.

Profile 1: claude-sdk

  • Entrypoint: node claude-runner.js
  • Behavior: Initializes the SDK locally. Real-time streaming. The container is the thinking entity.

Profile 2: remote-bridge

  • Entrypoint: node bridge-runner.js --provider=...
  • Behavior:
    1. Connects to Cloud API.
    2. Polls/Streams for events.
    3. CRITICAL: Executes "Function Calls" (tools) inside the container and reports results back to cloud.

4. Real-Time Deployment Strategy

The "Warm Pool"

To avoid Docker startup latency (5-10s):

  1. Pre-Boot: Orchestrator maintains ~3 idle LiquidContainer instances.
  2. Assignment: New task -> Grab idle container -> Send /init payload (~10ms).
  3. Replenishment: Asynchronously spin up new containers to refill pool.

Dependency Management ("The Heavy Base")

We cannot run npm install at runtime.

  • Base Image: Pre-installed Bun, Node.js 20, TypeScript, Git, Python.
  • Code Mount: Host project /Users/mario/projects/LiquidCrypto is mounted to /app (Read-Only/Overlay) so agents always see the latest code.

5. Implementation Design

Dockerfile

FROM oven/bun:latest as base

# Install Node compatibility & Tools
RUN apt-get update && apt-get install -y nodejs npm git iproute2

# Install SDKs Globally
RUN npm install -g @anthropic-ai/claude-agent-sdk
RUN pip install claude-agent-sdk

# Setup App Directory
WORKDIR /app

# Runtimes
COPY runtimes/sdk-wrapper.ts /usr/local/bin/
COPY runtimes/raw-server.ts /usr/local/bin/

# Default Entrypoint (Overridden by Orchestrator)
CMD ["bun", "run", "/usr/local/bin/raw-server.ts"]
